# ADR-002: Use DESIGN.md As Visual Design Source Of Truth

## Status

Accepted

## Context

The repo now has a root `DESIGN.md` that defines visual identity tokens and guidance for UI implementation. Previously, runtime style files such as `src/app/styles.css` and `src/components/Button/Button.module.css` were treated as the practical source for colors, typography, spacing, and component styling.

That creates drift risk: agents and humans can update runtime CSS without updating the design source, or update `DESIGN.md` without changing the app.

## Decision

`DESIGN.md` is the authoritative source of truth for visual design decisions in this repo.

Runtime style files are derived consumers of `DESIGN.md`, not independent sources of truth. This includes:

- `src/app/styles.css`
- `src/components/Button/Button.module.css`
- future component CSS modules or style files that express colors, typography, spacing, radius, borders, focus states, status colors, or component visual states

Generated token files may be committed, but they must be reproducible from `DESIGN.md`.

## Consequences

- Visual tokens must be changed in `DESIGN.md` first.
- Runtime CSS must consume generated CSS variables or generated component tokens.
- Direct hard-coded visual values in migrated UI files are not allowed unless documented as one-off exceptions.
- A drift check must fail verification when generated token files do not match `DESIGN.md`.
- Material changes to `DESIGN.md` follow the protected-path approval process.
- Existing runtime files will be migrated incrementally to generated tokens.

## Implementation Plan

1. Add a token generation script that reads `DESIGN.md`.
2. Generate committed CSS token files.
3. Import generated global tokens into the app.
4. Migrate `src/app/styles.css` and `src/components/Button/Button.module.css` to use generated variables.
5. Add `npm run design:tokens:check`.
6. Wire the drift check into `make verify`.

## Alternatives Considered

### Keep DESIGN.md manually synchronized with runtime CSS

Rejected because it still allows runtime styles and design guidance to drift unless every change is manually synchronized.

### Keep runtime CSS as source of truth

Rejected because runtime CSS is implementation-specific and less useful for AI agents, review, cross-repo standards, and design tooling.

## Verification

The decision is complete when:

- `repo-contract.yaml` declares generated visual identity outputs
- generated token files exist and are committed
- migrated runtime CSS uses generated variables
- `npm run design:tokens:check` passes
- `make verify` passes
